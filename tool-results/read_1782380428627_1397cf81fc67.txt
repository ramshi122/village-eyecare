     2тЖТ

     4тЖТ
     5тЖТ**NOT for:** Text-heavy flowing documents (guides, handbooks, catalogs, introductions) тЖТ use `briefs/creative-flow.md` instead.

     8тЖТ

    22тЖТ## Phase 1: The Editorial Eye (Content Transformation)
    23тЖТ
    24тЖТBefore you design, you must "edit the raw material". Users often provide dense, unstructured text. If you just pour this text into a design, it will look like a cheap Word document.
    25тЖТ
    26тЖТYou must apply **Editorial Pacing**:
    27тЖТ
    28тЖТ### 1. The Word Budget
    29тЖТNo single page or visual canvas should exceed **150 words** of readable body text (the golden rule for maximum aesthetic impact). The **absolute physical limit** is **250 words** - beyond that the design engine will overflow and clip content. If the raw content exceeds 150 words:
    30тЖТ- **Action A**: Brutally summarize it down to тЙд150 words.
    31тЖТ- **Action B**: Split it across multiple `pages` in your JSON blueprint.
    32тЖТ- Only push to 150-250 words if the content genuinely cannot be cut further.
    33тЖТ
    34тЖТ### 2. Typographic Hierarchy Extraction
    35тЖТYou must parse the raw text and categorize it into typographic roles:
    36тЖТ- **Hero / Display**: The core emotional hook (1-5 words). Must be punchy.
    37тЖТ- **Kicker / Eyebrow**: Tiny context text above a headline (e.g., "Q3 REPORT", "MANIFESTO").
    38тЖТ- **Lead Paragraph**: The 2-3 sentence summary.
    39тЖТ- **Data Sculptures**: Scan the text for impactful numbers (e.g., "97%", "$4.2M"). EXTRACT THEM. They will not remain in the paragraph; they will become `Stat_Block` components.
    40тЖТ- **Pull Quotes**: Extract the most provocative sentence to stand alone as a visual anchor.

    44тЖТ## Phase 2: The Component Lexicon
    45тЖТ
    46тЖТYou will construct your JSON Blueprint using ONLY the following components. These map directly to the `configs/components.md` assets. Do not invent new component types.
    47тЖТ
    48тЖТ> **тЪая╕П Markdown Content Limits**: For `Glass_Canvas`, the golden rule is **under 150 words** for maximum aesthetic breathing room. Absolute physical limit is **250 words**. If content exceeds 250 words, the design engine will overflow. You MUST summarize it or split it into a new page.
    49тЖТ
    50тЖТ### 1. `Hero_Typography`
    51тЖТGiant, page-dominating text. Usually interacts with the background via blend modes.
    52тЖТ- **Parameters**:
    53тЖТ  - `content` (string): The text (use `<br>` for deliberate line breaks).
    54тЖТ  - `weight` (string): `"black"` (900, dominating) or `"thin"` (100, elegant).
    55тЖТ  - `variant` (string, optional): `"standard"` only. ~~`"vertical_accent"`~~ is **NOT implemented** in `design_engine.py` - the engine silently ignores this value and renders nothing. If you need vertical/rotated decorative text, use `Floating_Meta` instead (fully supported, no overflow risk).
    56тЖТ  - `scale` (integer, optional): Typographic scale level `1`-`6`. Controls font size via the engine's fluid type system. If omitted, the engine uses the default hero size.
    57тЖТ    - `6`: Hero/Display - oversized title, clamp(64px, 12vw, 150px). Use for single words or short phrases with maximum visual impact.
    58тЖТ    - `5`: Primary Title - clamp(48px, 8vw, 96px). Standard poster headline.
    59тЖТ    - `4`: Subheadline - clamp(32px, 5vw, 56px). Chapter openers or key quotes.
    60тЖТ    - `3`: Lead Paragraph - clamp(20px, 3vw, 32px). Prominent body text.
    61тЖТ    - `2`: Body - 16px. Standard readable text.
    62тЖТ    - `1`: Meta/Caption - 10px. Decorative, environmental.
    63тЖТ
    64тЖТ### 2. `Glass_Canvas`
    65тЖТThe main structural container for reading text. Frosted glass, sharp 2px print corners.
    66тЖТ- **Parameters**:
    67тЖТ  - `markdown_content` (string): The actual text to read. Supports standard Markdown (H2, H3, bold, lists). *Must be under 150 words.*
    68тЖТ  - `tension_score` (float, optional): Semantic tension value from `0.0` to `1.0`. Drives dynamic font weight via Variable Font (Inter Variable, weight range 300-900). The engine maps: `weight = 300 + (tension_score ├Ч 600)`.
    69тЖТ    - `0.0-0.2` тЖТ Light (300-420): calm, contemplative passages
    70тЖТ    - `0.3-0.5` тЖТ Normal (420-600): standard body text
    71тЖТ    - `0.6-0.8` тЖТ Bold (600-780): urgent, assertive content
    72тЖТ    - `0.9-1.0` тЖТ Max (780-900): crisis, climax, emotional peak
    73тЖТ    - **When to use**: Multi-page narrative documents with emotional arc (case studies, pitch decks, manifestos). Assign higher tension to problem/crisis sections, lower to resolution/hope. Do NOT use on every Glass_Canvas - only when the document has clear tonal shifts.
    74тЖТ    - **When NOT to use**: Data-heavy pages, simple reports, single-page posters.
    75тЖТ
    76тЖТ### 3. `Floating_Meta`
    77тЖТTiny, environmental metadata (dates, edition numbers, catalog IDs) that lives in the 15% breathing margin.
    78тЖТ- **Parameters**:
    79тЖТ  - `position` (string): `"top-left"`, `"top-right"`, `"bottom-left"`, `"bottom-right"`.
    80тЖТ  - `items` (array of strings): E.g., `["VOL 01", "2026", "EDITION 500"]`.
    81тЖТ
    82тЖТ### 4. `Hairline_Divider`
    83тЖТStructural 0.5px line. Not decorative; acts as a visual fold.
    84тЖТ- **Parameters**:
    85тЖТ  - `style` (string): `"bleed"` (goes edge-to-edge) or `"accent"` (short 30% width line).
    86тЖТ
    87тЖТ### 5. `Stat_Block`
    88тЖТData sculpture. A massive number with a tiny label.
    89тЖТ- **Parameters**:
    90тЖТ  - `number` (string): e.g., `"97.3"`.
    91тЖТ  - `unit` (string): e.g., `"%"`, `"Hz"`, `"$M"`.
    92тЖТ  - `label` (string): e.g., `"COMPLETION RATE"`.
    93тЖТ
    94тЖТ### 6. `Image_Asset`
    95тЖТA visual element. The engine will apply a gradient blend to it.
    96тЖТ- **Parameters**:
    97тЖТ  - `source` (string): A URL, or a descriptive prompt if you expect the system to generate it.
    98тЖТ  - `caption` (string, optional): Tiny text under the image.
    99тЖТ
   100тЖТ> тЪая╕П **Image_Asset is for CONTENT images only** (user-provided photos, logos, diagrams, charts). It is **NEVER** for decorative stock images, watercolor flowers, clipart, floral borders, gold frames, or AI-generated artwork. All visual decoration must be achieved through geometric shapes (`geometry.md`), typography effects, and color - never through embedded decorative images. See `visual_framework.md` Stock Image Ban.
   101тЖТ
   102тЖТ### 7. `Page_Ghost_Number`
   103тЖТA giant, 4% opacity number acting as a watermark in the background.
   104тЖТ- **Parameters**:
   105тЖТ  - `number` (string): e.g., `"01"`, `"X"`.
   106тЖТ
   107тЖТ### 8. `Delta_Widget`
   108тЖТ**Data-to-Ink Ratio enforcer.** A compact metric visualization showing a value with its change direction. CRITICAL: Use this instead of writing sentences like "revenue grew by 12%". Extract every trend into a Delta_Widget.
   109тЖТ- **Parameters**:
   110тЖТ  - `metric` (string): The metric name, e.g., `"REVENUE"`, `"LATENCY"`.
   111тЖТ  - `value` (string): Current value, e.g., `"$4.2M"`, `"45ms"`.
   112тЖТ  - `delta` (string): Change description, e.g., `"+12%"`, `"-45ms"`.
   113тЖТ  - `trend` (string): `"up"`, `"down"`, or `"flat"`.
   114тЖТ  - `label` (string, optional): Context line, e.g., `"vs. Q2 2025"`.
   115тЖТ
   116тЖТ### 9. `Process_List`
   117тЖТ**Polymorphic adaptive component.** Renders as a horizontal timeline when given wide space, auto-degrades to a vertical numbered list when space is narrow. Use for workflows, steps, timelines.
   118тЖТ- **Parameters**:
   119тЖТ  - `steps` (array): Each item has `title` (string) and `description` (string).
   120тЖТ
   121тЖТ### 10. `Sidenote_Block`
   122тЖТ**Tufte marginalia.** Used in `tufte_report` archetype layouts. Content is placed in the 30% side rail alongside the main column. Perfect for citations, supplementary data, asides, footnotes.
   123тЖТ- **Parameters**:
   124тЖТ  - `label` (string, optional): Category label, e.g., `"SOURCE"`, `"NOTE"`, `"DATA"`.
   125тЖТ  - `body` (string): The sidenote content in Markdown.
   126тЖТ
   127тЖТ### 11. `Data-Aware Background` (via `data_points`)
   128тЖТAny component can carry an optional `data_points` array (e.g., `[10, 15, 8, 24, 30]`). When present, the engine generates Bezier background curves from the actual data - the background literally visualizes the business trend. Use on pages with financial, metric, or time-series content.

   133тЖТ
   134тЖТThe following components are specialized generative design tools. They unlock visual effects that standard components cannot achieve. Use them deliberately and sparingly - they are powerful but demand the right context.
   135тЖТ
   136тЖТ### 8. `Shaped_Canvas`
   137тЖТA container where text flows around a non-rectangular shape. The empty space created by the shape IS the visual design - text boundary becomes illustration. Uses CSS `shape-outside` for non-rectangular text wrapping.
   138тЖТ
   139тЖТ- **Parameters**:
   140тЖТ  - `shape_keyword` (string): One of `"circle"`, `"wave"`, `"diagonal_slash"`, `"diamond"`, `"wedge_right"`.
   141тЖТ  - `markdown_content` (string): The text that will flow around the shape. Supports standard Markdown.
   142тЖТ
   143тЖТ- **Shape Selection Guide**:
   144тЖТ  | shape_keyword | Visual Effect | Thematic Fit |
   145тЖТ  |---------------|---------------|--------------|
   146тЖТ  | `"circle"` | Text wraps around a circular void on the left | Unity, spotlight, focus |
   147тЖТ  | `"wave"` | Wavy left text boundary | Ocean, flow, music, fluidity |
   148тЖТ  | `"diagonal_slash"` | Diagonal cut across the page | Disruption, change, transformation |
   149тЖТ  | `"diamond"` | Diamond-shaped negative space | Luxury, precision, crystalline |
   150тЖТ  | `"wedge_right"` | Arrow/wedge pointing right | Direction, progress, forward motion |
   151тЖТ
   152тЖТ- **When to use**:
   153тЖТ  - Artistic/editorial pages that need visual drama without images
   154тЖТ  - Cover or section opener pages where you want a "wow" moment
   155тЖТ  - When the content thematically maps to a recognizable shape
   156тЖТ  - Only on pages with moderate text (100-200 words) - shape eats 30-40% of space
   157тЖТ
   158тЖТ- **When NOT to use**:
   159тЖТ  - On data-heavy or dense content pages
   160тЖТ  - Together with `Glass_Canvas` on the same page (visual conflict)
   161тЖТ  - More than one `Shaped_Canvas` per page
   162тЖТ
   163тЖТ- **Archetype requirement**: Pages using `Shaped_Canvas` MUST use archetype `"shaped_editorial"` (relaxed safe-zone: 5% 6% inset).

   165тЖТ### Chart & Data Visualization Styling
   166тЖТ

   170тЖТ- **Donut > Pie**: Always use ring charts (hole ratio 60-70%), center area displays total/metric

   179тЖТ## Phase 3: Page Archetypes (The Grid Strategy)
   180тЖТ
   181тЖТFor each page in your JSON, you must declare an `archetype`. This tells `design_engine.py` how to arrange the components you provided.
   182тЖТ
   183тЖТ- `"cover_hero"`: Cover page. **Must follow `typesetting/cover.md` 5-template system** - pick Layout 01/03/04/06/07 based on document tone. See "Cover Page Constitution" below for iron rules.
   184тЖТ- `"split_vertical"`: Page split strictly 50/50 vertically. Left side image/svg, right side Glass_Canvas.
   185тЖТ- `"editorial_flow"`: Top-down reading experience. Centered columns, generous margins. Use for main content.
   186тЖТ- `"scattered_canvas"`: No grid. Elements placed via absolute positioning based on spatial weights.
   187тЖТ- `"data_dashboard"`: 2x2 or 3x3 strict grid for multiple `Stat_Block`s.
   188тЖТ- `"shaped_editorial"`: Relaxed safe-zone (5% 6% inset) designed for `Shaped_Canvas`. Centered, generous breathing room. **Must be used when the page contains a `Shaped_Canvas`.** Do NOT mix with `Glass_Canvas`.
   189тЖТ- `"tufte_report"`: **Tufte marginalia layout.** 70% main content column + 30% sidenote rail. Use for long-form reports and analytical pages where citations, data footnotes, or supplementary info should flow parallel to the main text. Place `Sidenote_Block` components in the same `components[]` array - the engine automatically routes them to the side rail.
   190тЖТ
   191тЖТ### тШЕ 12├Ч12 CSS Grid Coordinate System (Iron Rules)
   192тЖТ
   193тЖТThe layout engine uses a **12-track CSS Grid** for element placement. The grid lines are numbered **1 to 13** (12 tracks = 13 lines).
   194тЖТ
   195тЖТ**Absolute boundary rules - violation = broken layout:**
   196тЖТ- **Column lines**: `1` = absolute left edge, `13` = absolute right edge. Full width = `1 / 13`.
   197тЖТ- **Row lines**: `1` = absolute top edge, `13` = absolute bottom edge. Full height = `1 / 13`.
   198тЖТ- **CRITICAL**: Never output a grid line number less than `1` or greater than `13`. Any value outside `[1, 13]` will destroy the layout.
   199тЖТ
   200тЖТ**`grid_area` format**: `"row_start / col_start / row_end / col_end"` (CSS shorthand).
   201тЖТ- Example: `"1 / 1 / 7 / 13"` = top half of the page, full width.
   202тЖТ- Example: `"3 / 8 / 6 / 13"` = rows 3-5, columns 8-12 (right side block).
   203тЖТ
   204тЖТ**40% whitespace rule**: At least 40% of grid cells (тЙе58 out of 144) must be left empty. Count your occupied cells.

   206тЖТ### тШЕ Cover Page Constitution (5 Layout System)
   207тЖТ
   208тЖТCover pages (`archetype: "cover_hero"`) are the first impression. They must be ruthlessly sparse and spatially sophisticated.
   209тЖТ
   210тЖТ**тЖТ MUST READ: `typesetting/cover.md`** before designing any cover.
   211тЖТ
   212тЖТ> **тЪая╕П Template Numbering:** Cover HTML templates use `cover.md`'s authoritative numbering: **01** (HUD), **03** (Academic Anchor), **04** (Academic Symmetric), **06** (Institutional), **07** (Crystal Blue). The "7 Cover Layouts" table below (1AтАУ7B) describes **Blueprint grid coordinate patterns** for positioning cover elements, NOT the HTML template IDs. Do not confuse the two systems.
   213тЖТ
   214тЖТ#### Global Iron Rules (Always Apply)
   215тЖТ
   216тЖТ1. **Maximum 4 components** on any cover page. Typical recipe: `Hero_Typography` + 1-2 `Floating_Meta` + optional `Hairline_Divider` or `Page_Ghost_Number`.
   217тЖТ2. **Typography Scale**: Title тЙИ 45pt (2.5├Ч base), Subtitle тЙИ 25pt (1.4├Ч base), Meta тЙе 18pt (never below 14pt). Covers with tiny text = FAIL.
   218тЖТ3. **Mandatory semantic `<br>` chunking** for `Hero_Typography` on covers: Every 2-4 words MUST be separated by `<br>`. Single-line hero text is FORBIDDEN on covers. Example: `"ALGORITHMIC<br>FATIGUE"`, NOT `"ALGORITHMIC FATIGUE"`.
   219тЖТ4. **Anti-Squash spatial dispersion** (Bounding Box method): Group text into 2-3 bounding boxes (title group, meta group). Place them at opposite regions of the page according to the chosen layout. Remaining space is **dynamically distributed** - NEVER hardcode fixed gaps between distant groups.
   220тЖТ5. **No `Glass_Canvas` on cover pages.** Dense reading text kills the visual impact. Push all body content to page 2+.
   221тЖТ6. **Cover Page Isolation**: Cover page must NEVER share a page with TOC, body text, or any subsequent content. The cover is always a standalone full page. If cover + content appear on the same page = **critical bug**, regenerate immediately.
   222тЖТ7. **Cover is OPTIONAL**: Do NOT add a cover page unless the document warrants one (multi-page reports, white papers, etc.) or the user explicitly requests it. Short documents, letters, memos, forms, and quick outputs skip the cover.
   223тЖТ8. **Background Layer (optional)**: See `typesetting/cover-backgrounds.md` for 3 recipes - A (цЮБчоАх╝зч║┐), B (х╖ечиЛхНБхнЧш╜┤+члЛцЯ▒), C (щФРшзТхИЗхЙ▓+хЗ║шбАцЦЗхнЧ). Background renders BELOW all foreground at 2-5% opacity. Pick a recipe that matches the document tone. Never combine elements across recipes.

   256тЖТ### тШЕ Component Grid Compatibility Constraints
   257тЖТ
   258тЖТWhen placing advanced components into the 12├Ч12 grid, obey these minimum size rules:
   259тЖТ
   260тЖТ- **`Glass_Canvas`**: Must occupy at least **6 columns ├Ч 4 rows** (e.g., `"3 / 1 / 7 / 7"`). Smaller areas will cause text overflow and padding collapse. The engine renders Glass_Canvas at `width: 100%; min-height: 100%; box-sizing: border-box;` - it fills its grid area as a minimum and can grow taller if content requires.
   261тЖТ- **`Shaped_Canvas`**: Must occupy at least **8 columns ├Ч 6 rows** (e.g., `"2 / 3 / 8 / 11"`). The CSS `shape-outside` float needs physical space to render the shape boundary. Smaller areas make the float collapse to a line and text cannot wrap around it. **Must use archetype `"shaped_editorial"`.**
   262тЖТ- **`Continuous Flow` background**: Renders in a separate `bg-layer` (`z-index: 1`, `position: absolute`), physically isolated from the grid system (`z-index: 2`). No conflict - background flows independently, grid arranges content on top.
   263тЖТ
   264тЖТ### тШЕ Content-Proportional Grid Row Allocation
   265тЖТ
   266тЖТWhen **two or more text-heavy components** (e.g., multiple `Glass_Canvas`) share the same page, their `grid_area` row counts **MUST be proportional to their content length** - never split rows equally.
   267тЖТ
   268тЖТ**Estimation method:**
   269тЖТ1. Count characters (or words) in each component's `markdown_content`
   270тЖТ2. Allocate rows proportionally: `rows_i = total_rows ├Ч (chars_i / total_chars)`
   271тЖТ3. Round to integers, minimum 3 rows per component
   272тЖТ
   273тЖТ**Example:**
   274тЖТ```
   275тЖТAttractions content: 450 chars (3 items ├Ч 150 chars)
   276тЖТFood content: 250 chars (2 items ├Ч 125 chars)
   277тЖТTotal available rows: 8 (rows 5тЖТ13)
   278тЖТ
   279тЖТAttractions: 8 ├Ч 450/700 тЙИ 5 rows тЖТ grid_area "5 / 1 / 10 / 13"
   280тЖТFood:        8 ├Ч 250/700 тЙИ 3 rows тЖТ grid_area "10 / 1 / 13 / 13"
   281тЖТ```
   282тЖТ
   283тЖТ**Why this matters:** Equal row allocation (4+4) causes the longer component to overflow into the shorter one's territory, creating text overlap in the final PDF. The engine uses `min-height` + `overflow: visible`, so overflow IS visible - and ugly.
   284тЖТ
   285тЖТ**Anti-pattern to avoid:**
   286тЖТ```
   287тЖТтЭМ Two Glass_Canvas with very different text lengths both get "5/1/9/13" and "9/1/13/13"
   288тЖТтЬЕ Longer one gets more rows: "5/1/10/13" and "10/1/13/13"
   289тЖТ```

   297тЖТ
   298тЖТ### The JSON Schema Specification
   299тЖТ
   300тЖТ**CRITICAL JSON RULES:**

   371тЖТ

   373тЖТ   - **`"auto"`: (RECOMMENDED DEFAULT). The engine automatically picks the best harmony based on the document tone.** Only override if you have a specific artistic reason.
   374тЖТ   - `"complementary"`: (180┬░ opposite). Strong visual contrast. Cold background with warm accent. For striking/dynamic impact.
   375тЖТ   - `"split_complementary"`: (150┬░/210┬░). Highly sophisticated, artistic, luxury feel (Editorial/Kinfolk style).
   376тЖТ   - `"analogous"`: (30┬░ apart). Harmonious, peaceful, natural transition.
   377тЖТ   - `"triadic"`: (120┬░ apart). Rich and slightly retro.
   378тЖТ   - `"monochrome"`: Only use if strict minimalist corporate branding without accent is required.

   630тЖТ
   631тЖТBefore you output the JSON block, verify:
   632тЖТ0. ** VECTOR OUTPUT IRON RULE:** The final PDF MUST be generated via `page.pdf()` / `convert.blueprint` - NEVER `page.screenshot()` тЖТ image тЖТ wrap as PDF. Screenshot PDFs are blurry raster images. `page.pdf()` produces vector text that stays sharp at any zoom level.
   633тЖТ1. **Did I write ANY HTML or CSS?** If yes, delete it. Only output JSON.
   634тЖТ2. **Is any `Glass_Canvas` markdown content too long?** Count the words. Over 150? Summarize it or push to the next page.
   635тЖТ3. **Is the JSON perfectly formatted?** Missing commas or unescaped quotes will crash the `design_engine.py` parser.
   636тЖТ4. **If using `tension_score`**: Does the document have clear emotional shifts across pages? If all pages are the same tone, remove it.
   637тЖТ5. **If using `continuous_flow`**: Is this multi-page (2+)? If single-page, switch to `"flow"`.
   638тЖТ6. **If using `Shaped_Canvas`**: Is the archetype `"shaped_editorial"`? Is there at most one per page? No `Glass_Canvas` on the same page?
   639тЖТ7. **Grid boundary check**: Are ALL `grid_area` values within `[1, 13]`? Any number < 1 or > 13 = FATAL ERROR.
   640тЖТ8. **Component minimum size check**: `Glass_Canvas` тЙе 6col├Ч4row? `Shaped_Canvas` тЙе 8col├Ч6row?
   641тЖТ9. **Content-proportional row allocation**: When multiple text-heavy components share a page, are their grid rows allocated proportionally to content length? Equal rows for unequal content = text overlap. (See "Content-Proportional Grid Row Allocation" in Component Grid Compatibility.)
   642тЖТ9. **40% whitespace check**: Count occupied grid cells. At least 58 of 144 cells must be empty.
   643тЖТ10. **Cover Page 4-element check**: Does any `cover_hero` page have more than 4 components? If yes, remove or merge components.
   644тЖТ11. **Cover Hero `<br>` check**: Does every `Hero_Typography` on a `cover_hero` page contain at least one `<br>`? Single-line hero text on covers = FAIL.

   682тЖТnode "$PDF_SKILL_DIR/scripts/html2pdf-next.js" input.html --output output.pdf --width 210mm --height 297mm
   683тЖТ```

   685тЖТ0. **Full-Bleed CSS (MANDATORY)**: Every HTML file for Playwright PDF MUST include:
   686тЖТ   ```css
   687тЖТ   @page { size: 800px 1200px; margin: 0; }  /* CONCRETE values only - CSS variables NOT supported in @page */
   688тЖТ   html, body { margin: 0; padding: 0; width: 800px; }
   689тЖТ   .page { width: 800px; height: 1200px; position: relative; }
   690тЖТ   ```
   691тЖТ   **тЪая╕П @page rules do NOT resolve CSS variables** (`var(--x)` is silently ignored, falls back to A4). Always use concrete `px` values.
   692тЖТ   - Each `.page` div = one PDF page, with fixed `height` matching `@page` size.
   693тЖТ   - **No `overflow: hidden`** on `.page` тАФ instead, constrain all decorative elements within page bounds (see ┬з0.75).
   694тЖТ   - Content must fit within each `.page` тАФ no cross-page flow.
   695тЖТ   - `design_engine.py` handles this automatically via `override_css`.
   696тЖТ
   697тЖТ0.25. **Body Background for Multi-Page Mixed-Color Documents (MANDATORY)**:
   698тЖТ   When an HTML document contains multiple `.page` divs with **different background colors** (e.g. dark cover + white body + dark specs page), Playwright's sub-pixel rounding creates <1px gaps at `.page` edges where `body` background shows through. On dark pages, a white `body` = visible white edges.
   699тЖТ
   700тЖТ   **Fix: set `html, body { background }` to the document's darkest page background color.**
   701тЖТ
   702тЖТ   ```css
   703тЖТ   :root { --primary: #0f172a; }  /* darkest page color */
   704тЖТ   html, body {
   705тЖТ     margin: 0; padding: 0;
   706тЖТ     width: 210mm;                 /* match @page size */
   707тЖТ     background: var(--primary);   /* fills sub-pixel gaps with dark color */
   708тЖТ   }
   709тЖТ   ```
   710тЖТ
   711тЖТ   **Why this doesn't break white pages:** White `.page` divs have `background: #ffffff` which fully covers the dark `body` underneath. The dark body only shows through the <1px sub-pixel gap at the extreme page edge - imperceptible on white pages after anti-aliasing.
   712тЖТ
   713тЖТ   **Selection rule:**
   714тЖТ   - All pages same color тЖТ `body { background: <that color> }`
   715тЖТ   - Mixed dark + light pages тЖТ `body { background: <darkest page color> }` (dark edges on white pages are invisible; white edges on dark pages are the bug we're fixing)
   716тЖТ   - All pages white/light тЖТ `body { background: <lightest content bg> }` (e.g. `#f8fafc`)
   717тЖТ
   718тЖТ0.5. **No overflow:hidden anywhere + Browser Preview Adaptive Scaling (MANDATORY)**:
   719тЖТ   **Never** set `overflow: hidden` on `html`, `body`, `.page`, or any container. Reasons:
   720тЖТ   - It silently clips content that slightly exceeds bounds, making layout bugs invisible.
   721тЖТ   - `html2pdf-next.js`'s pre-render check detects `scrollHeight > clientHeight` + `overflow: hidden` and triggers auto-fix (force-expanding the container), which may break the layout.
   722тЖТ   - Instead, constrain decorative elements within page bounds at the CSS level (see ┬з0.75).
   723тЖТ
   724тЖТ   **`design_engine.py` handles this automatically**: During blueprint compilation, it auto-injects `@media screen` centering + scaling code, and the `html` background color uses `var(--c-bg)` matching the poster's main color. No manual addition needed.
   725тЖТ
   726тЖТ   **Correct approach when writing HTML manually**: Remove `overflow: hidden` and manually add `@media screen` scaling preview:
   727тЖТ   ```css
   728тЖТ   /* Fixed canvas size, no overflow */
   729тЖТ   html, body { margin: 0; padding: 0; width: 800px; height: 1400px; }
   730тЖТ   .page { width: 800px; height: 1400px; position: relative; }
   731тЖТ
   732тЖТ   /* Auto-scale to viewport in browser preview for full-page view */
   733тЖТ   @media screen {
   734тЖТ     html {
   735тЖТ       height: auto;
   736тЖТ       display: flex;
   737тЖТ       justify-content: center;
   738тЖТ       background: #0a0a1a; /* Surround color тАФ must match poster's main background */
   739тЖТ     }
   740тЖТ     body {
   741тЖТ       transform-origin: top center;
   742тЖТ       scale: min(1, calc(100vw / 800), calc(100vh / 1400)); /* Consider both width and height */
   743тЖТ       margin: 0 auto;
   744тЖТ       box-shadow: 0 0 60px rgba(0,0,0,0.3); /* Optional: shadow to distinguish canvas in preview */
   745тЖТ     }
   746тЖТ   }
   747тЖТ   ```
   748тЖТ   `@media screen` rules only apply in browser preview; `page.pdf()` uses print media and is unaffected.
   749тЖТ   **Every fixed-size HTML must include this `@media screen` adaptive code.**
   750тЖТ
   751тЖТ0.75. **Decorative Element Containment (MANDATORY тАФ replaces overflow:hidden)**:
   752тЖТ   **Do NOT use `overflow: hidden` on `.page` containers.** It silently clips content that slightly exceeds bounds, making layout errors invisible and debugging painful.
   753тЖТ
   754тЖТ   Instead, **constrain all decorative elements within page bounds at the CSS level**:
   755тЖТ
   756тЖТ   ```css
   757тЖТ   /* тЬЕ CORRECT: decorative element stays inside .page */
   758тЖТ   .deco-circle {
   759тЖТ       position: absolute;
   760тЖТ       width: 80%;           /* NOT >100% */
   761тЖТ       height: 40%;
   762тЖТ       top: 10%;
   763тЖТ       right: 0;             /* NOT negative */
   764тЖТ       border-radius: 50%;
   765тЖТ       opacity: 0.1;
   766тЖТ   }
   767тЖТ
   768тЖТ   /* тЭМ FORBIDDEN: causes scrollWidth > pageWidth тЖТ Playwright shrinks everything */
   769тЖТ   .deco-circle-bad {
   770тЖТ       position: absolute;
   771тЖТ       width: 200%;          /* inflates scrollWidth */
   772тЖТ       left: -50%;           /* extends beyond left edge */
   773тЖТ   }
   774тЖТ   ```
   775тЖТ
   776тЖТ   **Iron rules for decorative elements:**
   777тЖТ   - `width` тЙд 100% of `.page` width. No exceptions.
   778тЖТ   - `left` тЙе 0, `right` тЙе 0. No negative offsets that push content beyond page edges.
   779тЖТ   - For edge-bleeding effects, position the element at the edge and let `border-radius` or `opacity` create the visual bleed within bounds.
   780тЖТ   - For horizontal flex layouts (тЙе3 items), always add `flex-wrap: wrap` to prevent horizontal overflow.
   781тЖТ
   782тЖТ   **Why not overflow:hidden?**
   783тЖТ   PlaywrightтАЩs `page.pdf()` checks `body.scrollWidth`. If any absolutely-positioned child pushes `scrollWidth` beyond `@page` width, Playwright shrinks **all content** to fit. `overflow:hidden` masks this by clipping, but also silently hides any content that slightly exceeds bounds. Constraining elements at the source eliminates both the shrink bug and the silent clipping.
   784тЖТ
   785тЖТ   See `typesetting/overflow.md` ┬з3.5 for additional horizontal overflow rules.

   787тЖТ1. **Character Encoding Safety**: Never use Japanese kana (уБо, уБМ, уБп), rare symbols, or Private Use Area characters in content strings. They corrupt to U+FFFD (я┐╜) during LLMтЖТfile writeтЖТread transit. Replace with plain Chinese equivalents: `уБо`тЖТ`ф╣Л/чЪД/ч╝Ф/чЬБчХе`.
   788тЖТ2. **Vertical Chinese Text** - When using `writing-mode: vertical-rl` for CJK, you MUST include:
   789тЖТ   ```css
   790тЖТ   writing-mode: vertical-rl;
   791тЖТ   text-orientation: upright;    /* Each glyph stands upright */
   792тЖТ   white-space: nowrap;          /* Prevent word-wrap breaking single chars to new column */
   793тЖТ   letter-spacing: 12px;         /* Typical CJK vertical spacing */
   794тЖТ   ```
   795тЖТ   Without `text-orientation: upright`, Latin/fallback fonts render rotated 90┬░. Without `white-space: nowrap`, CJK characters may wrap into unexpected multi-column layouts (e.g., 3 chars on one line + 1 char alone on next).
   796тЖТ3. **Font Coverage**: For CJK content via Playwright, always load Google Fonts Noto Serif SC or Noto Sans SC via `<link>` tag in `<head>` (NOT `@import` in CSS - `@import` must be the very first rule in a stylesheet or it's silently ignored). Example:
   797тЖТ   ```html
   798тЖТ   <link href="https://fonts.googleapis.com/css2?family=Noto+Sans+SC:wght@300;400;500;700;900&display=swap" rel="stylesheet">
   799тЖТ   ```
   800тЖТ   System CJK fonts vary across macOS/Linux - Google Fonts guarantee glyph coverage without relying on system fonts. `design_engine.py` already handles this automatically via `<link>` tag.
   801тЖТ4. **Post-Generation Text Verification**: After Playwright renders the PDF, extract text from every page and scan for `?` or `\ufffd`. If found, the source HTML has encoding-corrupted characters that must be replaced in the Python source.
   802тЖТ5. ** HTML Pre-Render Validation (MANDATORY)**: After writing the HTML file and before running `html2pdf-next.js`, always run the HTML validator:
   803тЖТ   ```bash
   804тЖТ   python3 "$PDF_SKILL_DIR/scripts/poster_validate.py" check-html <your_file>.html
   805тЖТ   ```
   806тЖТ   - **ERROR** items (e.g. `OVERFLOW_HIDDEN_CONTAINER`, `FONT_NO_FALLBACK`) тЖТ must fix before PDF generation. Use `--fix --output <file>.html` for auto-repair.
   807тЖТ   - **WARNING** items (e.g. `FIXED_SIZE_NO_SCREEN_ADAPT`, `SCREEN_ADAPT_NO_SCALE`, `COLOR_CONTRAST`) тЖТ review and fix where appropriate.
   808тЖТ   - This catches the most common bypass HTML bugs: `overflow:hidden` on containers, missing `@media screen` auto-scale for fixed-size pages, font-family without generic fallback, low contrast text, etc.
